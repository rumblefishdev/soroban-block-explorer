# Add milestone field to backlog tasks and reorder by deliverable

## Summary

Backlog tasks are currently numbered by layer (DB → API → Frontend → Indexer → Infra), not by deliverable milestone. This means Deliverable 1 (Indexing Pipeline & Core Infra — 20% budget) cannot be completed first because its tasks (XDR parsing 0060–0066, CDK infra 0068–0078) are numbered at the end of the backlog. Add a `milestone` frontmatter field to all backlog tasks and update the index generator to group/sort by milestone, so the team works in deliverable order.

## Status: Active

**Current state:** Not started.

## Context

The technical design (`docs/architecture/technical-design-general-overview.md` §7.4) defines three deliverables:

| Deliverable                             | Scope                                                                                    | Budget |
| --------------------------------------- | ---------------------------------------------------------------------------------------- | ------ |
| **D1** — Indexing Pipeline & Core Infra | Galexie ECS, Lambda Processor, XDR parsing, DB core tables, CDK infra, CI/CD, CloudWatch | 20%    |
| **D2** — Complete API + Frontend        | All REST endpoints, React SPA, event interpretation, caching                             | 30%    |
| **D3** — Mainnet Launch                 | Production deploy, tests, load tests, security audit, monitoring                         | 40%    |

Current task numbering puts D1 infrastructure tasks (0060–0078) after all D2 tasks (0023–0059). Working in ID order would complete D1 last instead of first.

### Problem

Task IDs are stable identifiers used in `related_tasks` references, git commit scopes (`lore-NNNN`), and archived tasks. Renumbering would break ~90+ cross-references and git history traceability. Instead, a `milestone` field decouples ordering from identity.

## Implementation Plan

### Step 1: Define milestone mapping

Classify all backlog tasks into deliverables based on technical design §7.4:

**Milestone 1 (D1 — Indexing Pipeline & Core Infra):**

- 0015–0022 (DB schema + Drizzle + migrations + partitions)
- 0023 (NestJS scaffold — core only)
- 0038 (OpenAPI docs portal — design lists "OpenAPI specification" in D1 scope)
- 0060–0066 (XDR parsing + Indexer)
- 0068–0076, 0078 (CDK infra + CI/CD)

**Milestone 2 (D2 — Complete API + Frontend):**

- 0024–0037 (Backend API modules + cache)
- 0039–0059 (Frontend UI + pages)
- 0067 (Event interpreter)
- 0077 (MUI theme)

**Milestone 3 (D3 — Mainnet Launch):**

- No backlog tasks yet (testing, security, load test tasks to be created later)

### Step 2: Add `milestone` field to all task frontmatter

Add `milestone: 1` or `milestone: 2` after the `tags` field in each task's YAML frontmatter (backlog + active):

```yaml
tags: [priority-high, effort-medium, layer-database]
milestone: 1
```

### Step 3: Update lore README.md status table to sort by milestone then ID

The backlog board is generated by `lore-framework-mcp` (`index-generator.js` → `generateStatusTable()`). Currently it sorts by status order then ID. Change the sort to: **status → milestone → ID**.

This affects:

- `generateStatusTable()` — the Task Status table in `lore/README.md`
- `parseTasks()` — must read the `milestone` field from frontmatter

The generator lives in the `lore-framework-mcp` npm package (`dist/tools/index-generator.js`). This requires an upstream change to the package.

**Note:** `next-tasks.md` only shows active/blocked "ready" tasks (not backlog), so it does not need milestone sorting — the README.md status table is where backlog ordering matters.

### Step 4: Write ADR

Document the decision to use `milestone` field for deliverable ordering (instead of renumbering task IDs) as a formal ADR in `lore/2-adrs/`.

## Acceptance Criteria

- [ ] All backlog and active tasks have `milestone: N` field in frontmatter (N = 1, 2, or 3)
- [ ] Milestone assignments match the Three-Milestone Delivery Plan from technical design §7.4
- [ ] `lore/README.md` status table sorts backlog tasks by milestone first, then by ID
