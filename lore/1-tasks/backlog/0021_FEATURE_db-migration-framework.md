# Database migration framework

## Summary

Set up the Drizzle Kit migration framework with CDK deployment integration, environment-aware database targeting, and CI validation. Migrations must complete successfully before new Lambda code (API and Indexer) is deployed -- migration failure blocks deployment.

## Status: Backlog

**Current state:** Not started.

## Context

The block explorer uses Drizzle ORM for database access and Drizzle Kit for schema management and migration generation. The migration framework must work across three environments with different connection characteristics and must integrate into the CDK deployment pipeline as a hard prerequisite for application code deployment.

### Environment Handling

| Environment | Connection                | Migration Method                                      |
| ----------- | ------------------------- | ----------------------------------------------------- |
| dev         | Local PostgreSQL (direct) | Drizzle Kit CLI directly against local DB             |
| staging     | RDS through RDS Proxy     | Migration Lambda or CDK custom resource through proxy |
| production  | RDS through RDS Proxy     | Migration Lambda or CDK custom resource through proxy |

- In dev, migrations run directly via Drizzle Kit CLI commands (e.g., `drizzle-kit migrate`).
- In staging and production, migrations run through a dedicated migration step that connects through RDS Proxy, executed as part of the CDK deployment pipeline.

### CDK Integration Requirements

- Migrations MUST complete before deploying new Lambda code for both apps/api and apps/indexer.
- Migration failure MUST block the deployment -- no new application code is rolled out if the schema is not in the expected state.
- This can be implemented as a CDK custom resource, a pre-deployment Lambda, or a CodeBuild step within the CDK pipeline. The mechanism must guarantee ordering.

### Schema Evolution Rules

From the architecture documentation, schema changes must follow these rules:

- Add new tables or columns only when tied to a documented explorer or ingestion need.
- Never replace explicit relational structure with oversized generic JSON blobs.
- Keep public lookup keys stable where routes or API contracts depend on them.
- Update the general architecture overview first if the conceptual schema changes materially.

### Migration Versioning

- Migration files are generated by Drizzle Kit and committed to source control.
- Each migration is a versioned, ordered SQL file that represents an incremental schema change.
- CI validates that the current schema definitions match the generated migration state -- no drift is allowed between the Drizzle schema code and the migration files.

## Implementation Plan

### Step 1: Drizzle Kit configuration

Configure `drizzle.config.ts` at the workspace level with:

- Schema file paths pointing to the shared database library.
- Output directory for generated migration files.
- Database connection configuration that resolves per environment.

### Step 2: Migration directory structure

Establish the migration directory within the workspace. Migrations should live in a location that is:

- Version-controlled alongside the schema definitions.
- Accessible to both the local CLI workflow and the CDK deployment pipeline.

### Step 3: Local dev migration workflow

Set up the local development workflow:

- `drizzle-kit generate` to create new migration files from schema changes.
- `drizzle-kit migrate` to apply migrations to local PostgreSQL.
- `drizzle-kit push` for rapid iteration during development (direct schema push without migration files).

### Step 4: CDK migration integration

Implement the CDK deployment integration:

- Create a migration execution mechanism (custom resource Lambda or CodeBuild step).
- Wire it into the CDK deployment pipeline so it runs BEFORE Lambda function updates.
- Ensure the migration step connects through RDS Proxy for staging/production.
- Implement failure handling: if migration fails, the deployment is aborted.

### Step 5: CI validation

Add CI checks that:

- Run `drizzle-kit generate` and verify no new migration files are produced (schema and migrations are in sync).
- Optionally run migrations against a disposable PostgreSQL instance to validate they apply cleanly.

### Step 6: Rollback strategy

Document and implement the rollback approach:

- Drizzle Kit does not auto-generate down migrations. Rollback migrations must be written manually when needed.
- For non-destructive changes (adding columns/tables), rollback may not be necessary.
- For destructive changes, a manual rollback migration must be prepared and tested before deployment.

## Acceptance Criteria

- [ ] drizzle.config.ts is configured with correct schema paths and migration output directory
- [ ] Migration files are generated by Drizzle Kit and committed to source control
- [ ] Local dev workflow works: generate, migrate, and push commands function against local PostgreSQL
- [ ] CDK pipeline runs migrations before deploying new Lambda code
- [ ] Migration failure blocks deployment (no partial rollout of code without schema)
- [ ] Staging and production migrations connect through RDS Proxy
- [ ] CI validates that schema definitions and migration files are in sync
- [ ] Migration files apply cleanly to a fresh PostgreSQL instance in CI
