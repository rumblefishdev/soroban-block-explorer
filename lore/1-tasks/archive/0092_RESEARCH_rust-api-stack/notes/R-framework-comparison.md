## axum — Why Recommended

### Lambda Integration (Verified)

axum `Router` implements `tower::Service<http::Request>` — passes directly to `lambda_http::run()`. Zero adapter crate needed.

```rust
use axum::{routing::get, Router};
use lambda_http::{run, tracing, Error};

#[tokio::main]
async fn main() -> Result<(), Error> {
    tracing::init_default_subscriber();
    let app = Router::new()
        .route("/ledgers/{sequence}", get(get_ledger))
        .layer(CorsLayer::permissive())
        .with_state(state);
    run(app).await
}
```

Deps verified to compile together (`cargo check` passed, edition 2024):

- axum 0.8.8 + lambda_http 1.1.2 + sqlx 0.8.6 + utoipa 5.4.0 + utoipa-axum 0.2.0 + tower-http 0.6.8

No provisioned concurrency needed.

### Middleware

Tower-native. Full tower ecosystem: `tower-http` (CORS, tracing, compression), `tower` (timeout, rate limiting). Custom middleware via `from_fn`.

### OpenAPI

Two viable options evaluated:

**utoipa 5.4.0** (recommended — see R-openapi-crud-pagination.md for full comparison):

- Larger community (3,800 stars, 22.8M downloads)
- `utoipa-axum` 0.2.0 — stable, confirmed compatible with axum 0.8.8 (not stale, feature-complete)
- Better documentation quality: inline examples, `#[deprecated]` native, per-router tags
- Note: `#[aliases]` was removed in utoipa 5.x — use concrete types per resource (generated by `macro_rules!`)
- More annotation boilerplate (`#[utoipa::path(...)]` per handler) — mitigated by `macro_rules!` CRUD pattern
- 199 open issues but maintainer active (commits Feb 2026)
- **Verified:** `cargo check` passed with utoipa 5.4.0 + utoipa-axum 0.2.0 + axum 0.8.8 + sqlx 0.8.6 + lambda_http 1.1.2

**aide 0.15.1** (evaluated, rejected — see R-openapi-crud-pagination.md):

- Native axum 0.8 support, less CRUD boilerplate (pure generics, no macros)
- But: locked to **schemars 0.9** (transitional, current stable is **1.2.1**); no `#[deprecated]` support; no per-router tags; weak param examples; ugly generic schema names
- Smaller community (635 stars, 1.6M downloads)

**Recommendation: utoipa** — better documentation quality (examples, deprecation, schema naming, tags) outweighs aide's CRUD ergonomics advantage. See R-openapi-crud-pagination.md for full comparison.

## poem — Why Rejected (Despite Better OpenAPI DX)

### poem-openapi Advantage

poem-openapi is genuinely superior for OpenAPI generation:

- `#[OpenApi]` on impl block auto-generates spec from handler signatures
- `#[derive(Object)]` for schema, `#[derive(ApiResponse)]` for responses
- No duplicate annotations — handler IS the spec
- Built-in validation attributes

For 10 resources × 4-5 endpoints = ~50 endpoints, this saves significant boilerplate vs utoipa.

### Why Rejected

1. **Bus factor:** sunli829 authored 83% of commits (1,096/~1,320). No corporate backing. If maintainer stops → framework dies.
2. **No real-world Lambda+PG projects found.** Zero open-source poem + poem-lambda + sqlx + PostgreSQL examples. axum has at least vagmi/rust-lambda as exact-stack reference.
3. **poem-lambda has 190K total downloads** vs lambda_http's 5.8M. Tiny adoption for Lambda use case.
4. **189 open issues, last crate release July 2025** (8 months ago). Active repo pushes but no releases.
5. **Migration path exists** — if poem matures and axum disappoints, migration is ~2-3 days of HTTP layer rewrite.

## actix-web — Why Eliminated

1. **Lambda adapter dead:** `lambda-web` last GitHub push May 2024, last crate release January 2023. Not viable for production.
2. **Runtime incompatibility risk:** actix-rt (single-threaded tokio wrapper) introduces caveats with pure tokio crates. Unnecessary risk when axum runs on standard `#[tokio::main]`.
3. **No tower compatibility:** Own middleware trait (`Transform` + `Service`). Cannot use tower ecosystem.
4. **Declining adoption:** Community-driven since original author stepped back in 2020. New projects choosing axum.
